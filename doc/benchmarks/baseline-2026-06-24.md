### What each benchmark exercises (for interpreting deltas)

- **jam og / xeno** — `u3s_jam_fib` / `u3s_jam_xeno` on a small (~63-byte) ames
  `%work` packet noun, ×10 000. Dominated by the jam dedup hashtable + bitstream
  writer (report findings S1–S5).
- **jam cons / cons with** — jam of a large consed-up noun; `cons` builds the
  noun fresh each iter (so it also measures allocation/refcount churn),
  `cons with` reuses it. The **233 ms `jam cons`** is mostly noun construction +
  the loom-HAMT dedup path (report S4).
- **cue og / atom / xeno / test** — the several cue entry points on the jammed
  packet, ×10 000. `og` is the legacy noun-arithmetic cue (report S7 — slated
  for deletion); the others use the bitstream fast path.
- **cue cons / re-cons (~417 / ~596 ms)** — cue of a large jammed noun; the
  heaviest phases, dominated by allocation + the cue backref hashtable
  (report S1–S2, S6).
- **cue virtual og / atom** — cue run under `u3m_soft` (virtualized), measuring
  the soft-frame / road overhead on top of cue.
- **opcode 10 1k / 10k list** — `u3i_edit` (Nock opcode 10, edit) building a
  list by repeated edits; the **91 ms 10k** case is the only interpreter/edit
  hot path in the current suite. `1k` is sub-millisecond and not meaningful.

## Added benchmarks (branch `mw/benchmark_improvements`)

To cover the interpreter / dispatch / allocation paths the original suite
missed, four benchmarks were added to `benchmarks.c` (each builds its Nock
formula in C and **self-validates** before timing, so a mistake skips rather
than crashes). 10 runs, same build/host:

```
benchmark              min   med   mean  max   sd   exercises
-----------------------------------------------------------------------------
nock dec 200k loop      12  12.0   12.3   14  0.6   _n_burn + nock-9 kick (tail-ish loop)
ack(3,5)               106 106.5  106.9  111  1.4   deep doubly-recursive interp + frame stack
slam 500k               54  54.0   55.2   62  2.4   nock-9 kick / jet-dispatch per call
cons/free 100k x10      11  11.0   11.0   11  0.0   u3a_celloc + free + free-list reuse
reap/free 100k x10      11  11.0   11.4   12  0.5   u3qb_reap build + free
```

- **nock dec / ack** run a hand-built decrement and Ackermann formula via
  `u3n_nock_on`. `dec` is a tight loop; **`ack(3,5)` is deep non-tail
  recursion** (≈42k calls) that drives the interpreter frame stack — bump to
  `ack(3,6)`/`(3,7)` (≈170k/≈690k calls) for heavier stress.
- **slam** builds a trivial gate (`arm = +sample`) and slams it 500k times,
  isolating the per-call kick/dispatch cost (`u3j_kick` warm/cold + dispatch).
- **alloc** isolates the cell allocate/free fast path that `perf` flagged as the
  suite's dominant cost.

The original jam/cue/edit numbers above are unchanged by these additions
(verified across 10 runs — the new code only runs after them).

### Still not covered (need a pier/pill, out of scope for the lite harness)

- **Jetted dispatch with warm hits** — `slam`/`dec` exercise the kick path but
  against *unregistered* batteries (cold→fallback), because matching a real jet
  needs the Hoon-compiled battery from a pill.
- **Snapshot/save, boot, event replay** — need a full pier + event log; measure
  these with `boot-fake-ship.sh` / a pier-level harness, not this standalone
  executable.

## Environment limitations for profiling (this host)

- **`perf` is unavailable** here: `kernel.perf_event_paranoid = 4` and no
  passwordless sudo, so no hardware counters or sampling profile. On a host you
  control, `sudo sysctl kernel.perf_event_paranoid=1` re-enables it (see README).
- **`gprof` does not link**: the build is musl-static and musl provides no
  `mcount`, so `-pg` fails (`undefined symbol: mcount`). A glibc target would be
  needed for gprof.
- **Tracy** is the supported in-tree profiler and needs no kernel privileges for
  its instrumented zones — but capturing a trace headlessly needs the
  `tracy-capture` server tool, which is not installed here. The Tracy *build*
  path is verified working (`zig build benchmarks -Dtracy=true` builds and runs,
  identical timings); capture instructions are in `README.md`.

Net: this file is the authoritative **timing** baseline; the per-function
"where the time goes" baseline should be captured via Tracy (or `perf`) on a
host without these restrictions, using the same commit and build flags.
